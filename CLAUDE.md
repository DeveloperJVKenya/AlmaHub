# AlmaHub — Claude Code Project Guide

AlmaHub (package name `almahub`) is a Flutter + Firebase HR management system for JV Almacis, covering employee onboarding, recruitment, company policies, payroll, and leave management across five roles.

**Onboarding is HR-only.** Employees do not self-onboard — HR creates and submits every employee record via `lib/screens/hr/hr_employee_onboarding_screen.dart` (the 8-step wizard, `step1_personal_info.dart` … `step8_work_tools.dart`). An earlier employee self-onboarding flow existed but was eliminated; the Employee dashboard's own copy says "Contact HR to complete your onboarding" rather than offering a self-service form. Don't rebuild a self-onboarding entry point without confirming that's actually wanted — a prior one (`employee_onboarding_wizard.dart`) was removed as dead code because nothing referenced it.

## Roles & Navigation

Roles: **Admin, HR, Accountant, Supervisor, Employee** (`lib/models/user_model.dart` → `UserRoles`).

Login always routes to `lib/screens/employee/employee_dashboard.dart` first. That screen listens to the signed-in user's Firestore role (`_setupRoleListener`) and `Navigator.push`es HR/Supervisor/Accountant to their own dashboards; plain Employees stay on the dashboard itself, which doubles as their personal profile/self-service screen (view-only for onboarding data — see above). There is no central route table — every cross-screen navigation is an inline `Navigator.push(MaterialPageRoute(...))` at the call site.

- `lib/screens/hr/hr_dashboard.dart` — HR's main screen (employee directory, onboarding approvals, document management). Sub-features are NOT a Drawer/TabBar — they're `AppBar.actions` `IconButton`s that push standalone screens (Payroll, Policy Management, Employee Onboarding, Recruitment).
- `lib/screens/accountant/accountant_dashboard.dart` — **read-only** payroll report/export view. Accountant no longer creates or edits payroll; that's HR's job now (see Payroll below).
- `lib/screens/supervisor/supervisor_dashboard.dart` — shows the department's hours/overtime/performance table and reviews employee-submitted hours (see Hours workflow gotcha below); no longer where hours are directly entered by default.
- `lib/screens/employee/employee_dashboard.dart` — huge (3000+ line) side-menu-driven StatefulWidget; each menu item renders a different profile/self-service section (Personal Info, Employment, Statutory & Remuneration + Payslip History, Documents, Contracts, Work Tools, Leave).

## Feature-folder convention

Most role folders (`lib/screens/<role>/`) are flat files. The two most recently built HR features instead use a **self-contained feature folder**: `lib/screens/hr/<Feature Name>/` holding model(s) + service + document-actions helper + screen(s) together. Examples: `lib/screens/hr/Policy Management/` and `lib/screens/hr/Payroll/`. New HR features should follow this pattern, not the older flat-file one.

Note: `Policy Management` has a literal space in its folder name — Dart imports must URL-encode it as `Policy%20Management`.

## State management

The app is **StatefulWidget + setState + StreamBuilder/FutureBuilder directly against Firestore**, everywhere, except two deliberately scoped Riverpod islands: **`lib/screens/hr/Payroll/`** (state: selected month, run, records, settings) and the **employee/supervisor hours workflow** (`lib/services/hours_service.dart` + `lib/providers/hours_providers.dart`, state: selected month, daily entries, pending-review queue). `ProviderScope` wraps the app root in `main.dart` for this reason, but no other screen has been converted — don't refactor unrelated screens to Riverpod without discussing it first. Where either module's data needs to reach an otherwise-plain StatefulWidget, it's embedded via a scoped `Consumer`/`ConsumerWidget`, not a full screen conversion (e.g. `SupervisorDashboard` and `EmployeeHoursView` are `ConsumerStatefulWidget` specifically to host the hours providers; they don't use Riverpod for anything outside that).

## Firestore collections

- `Users` — Firebase Auth-linked user/role records.
- `EmployeeDetails` — every employee onboarding record (`EmployeeOnboarding` doc shape), draft or fully submitted — `status` (`draft`/`submitted`, plus legacy `approved`/`rejected` values that may still exist on old records) is the only completeness signal, there is no separate collection for in-progress records. There is no HR approve/reject action on this status anymore (see gotcha below) — `submitted` is the terminal state a record reaches once HR finishes filling it in. Carries `personalInfo`, `employmentDetails`, `statutoryDocs`, `payrollDetails` (standing salary/allowances/deductions/bank info, captured during onboarding step 4), `hoursWorked`/`daysWorked`/`workQuality` (Map<yyyy-MM, ...>, recomputed by `HoursService` from **approved-only** daily entries — see Hours workflow gotcha below), plus academic/contract/benefits/work-tools sections, and a `DailyHours` subcollection per employee (each doc has its own `status`: `pending`/`approved`/`rejected`, `source`: `employee`/`supervisor`).
- `Departments` — department docs; `members` map keys are employee UIDs.
- `CompanyPolicies` / `PolicyStatus` — Policy Management module.
- `PayrollSettings` (doc `config`), `PayrollRuns` (doc id = `yyyy-MM`), `PayrollRecords` (doc id = `{runId}_{employeeUid}`) — Payroll module. **Records are a snapshot at generation time** — editing an employee's `payrollDetails` later never mutates a historical payroll record.
- `LeaveData/{uid}` (+ `Applications` subcollection) — leave balances/applications.
- `Recruitees` — public job applications (unauthenticated writes allowed).

## Storage layout

`employee_documents/{employeeName}/{fieldName}/{filename}`, `company_policies/{filename}`, `payroll_payslips/{runId}/{employeeUid}.pdf`, `profile_photos/{userId}`, `Recruitees/{fileName}`.

## Conventions to follow when adding code

- **Models**: plain Dart classes, no codegen. `fromMap(map, [docId])` factory + `toMap()` + `copyWith()`. Firestore `Timestamp` ↔ `DateTime` conversion happens in the model, not the service.
- **Services**: one class per feature, private `CollectionReference`/`Reference` getters, every public method wrapped in try/catch using the shared `Logger` config below, returning a safe fallback (`[]`/`false`/`null`) instead of throwing to the UI.
- **Logger**: every service/screen instantiates its own `Logger(printer: PrettyPrinter(methodCount: 2, errorMethodCount: 8, lineLength: 120, colors: true, printEmojis: true, dateTimeFormat: DateTimeFormat.onlyTimeAndSinceStart))`. Match this exactly for consistency.
- **Document view/download**: reuse `lib/screens/hr/Policy Management/policy_document_actions.dart`'s pattern (or the thin Payroll wrapper around it) rather than reimplementing — it already handles web vs. native, office-viewer fallback, caching, and permissions. The underlying cross-platform save-to-disk logic lives in `lib/helpers/download_helper.dart` (`platformDownloadFile`), reusable as-is by any new feature.
- **Styling**: purple brand `Color.fromARGB(255, 86, 10, 119)` for headers/accents, white rounded-corner (8–12px) cards with a soft `BoxShadow`, `intl` `NumberFormat`/`DateFormat` for KES currency and dates.
- **Security rules**: `firestore.rules`/`storage.rules` gate almost everything on `request.auth != null` only — there's no custom-claims role check today, so app-level UI gating is what actually enforces HR/Admin-only vs. read-only access. Match this convention for new collections rather than introducing a heavier per-collection role-check pattern on its own.

## Known quirks / gotchas

- `lib/firebase_models.dart` is dead code (not imported anywhere) but documents an intended richer data model — useful as a reference, not as a dependency.
- **There used to be a separate `Draft` collection** for in-progress onboarding records, mirrored alongside `EmployeeDetails`. It was consolidated into a single `EmployeeDetails` collection (migrated via a one-time script, `scripts/migrate-draft-to-employeedetails/`) because completeness was being inferred from *which collection a document lived in* rather than checked directly — which let HR's onboarding wizard write incomplete records straight to "the complete collection" as long as it happened to be the last-visited step that validated cleanly. Don't recreate a `Draft` collection or any dual-collection fallback-query pattern (`try EmployeeDetails, fall back to Draft`); everything is `EmployeeDetails` filtered by `status` now.
- **Onboarding completeness is checked against the data, not the currently-mounted Form widget.** `hr_employee_onboarding_screen.dart`'s 8-step wizard only ever builds *one* step's `Form` at a time (`_buildStepContent` is a `switch`, not an `IndexedStack`), so `_formKeys[i].currentState` is `null` for every step except whichever is on screen. `lib/screens/hr/onboarding_completeness.dart`'s `OnboardingCompleteness` class checks the underlying step data models instead (mirroring each step's own `validator:` rules) and is the single source of truth for "is this record complete" — reused by both the wizard's Submit button and `PayrollService.generateRun`'s `isPayrollComplete` filter. Don't add a second, form-key-based completeness check.
- **Payroll eligibility is decided purely by payroll-field completeness** (`OnboardingCompleteness.isPayrollComplete` — basic salary + full bank details), independent of the employee's overall onboarding `status`. An employee still in `draft` status but with a fully-filled payroll step is included in payroll runs; a `submitted` employee with an empty payroll step is not. This was a deliberate product decision, not an oversight — don't add a `status != 'draft'` guard to `PayrollService.generateRun` without checking first.
- Payroll statutory constants (PAYE bands, NSSF, SHIF, Housing Levy) live in `lib/screens/hr/Payroll/payroll_statutory_calculator.dart` as `static const` — update there when Kenyan rates change, nowhere else.
- `BankDetails.branchName`/`accountName` are the current field names for the bank branch/account holder name. A small number of records written by the old, now-removed self-onboarding flow may still have data under the legacy key `branch` — `BankDetails.fromMap` reads `branchName` with a fallback to `branch`, and any new raw-map read of a bank details snapshot (e.g. `payroll_pdf_generator.dart`) should do the same rather than assuming one key.
- There used to be two near-duplicate onboarding-model files (`lib/models/employee_onboarding_models.dart` and `lib/screens/hr/employee_onboarding_models.dart`) that had quietly diverged — the HR onboarding flow wrote `branchName`/`accountName`/`trainingCertificates` while the canonical model didn't know about them. These have been consolidated into the single canonical `lib/models/employee_onboarding_models.dart`, which is now the one model every active read/write path uses (HR onboarding, HR dashboard, Payroll, Excel export, Employee dashboard display). Don't recreate a second copy under `lib/screens/hr/`.
- **Hours workflow: employee submits, supervisor approves, payroll reads only approved.** The old design had the supervisor directly type hours for an employee with no independent check. It's now: employee submits their own daily hours (`EmployeeHoursView`, status starts `pending`) → supervisor reviews and approves/rejects via `lib/screens/supervisor/hours_review_screen.dart` (confirm-dialog + colored-SnackBar pattern) → `HoursService._recomputeMonthlyTotals` only ever sums `status == 'approved'` days into the `EmployeeDetails.hoursWorked`/`daysWorked` maps that Payroll reads. A supervisor manual-entry fallback still exists (`supervisor_manual_hours_dialog.dart`, for employees without app access) but is written already-approved, always tagged `source: 'supervisor'` with a required justification note, and renders with an "Entered by Supervisor" badge everywhere — never mistaken for an independently-checked entry. Don't reintroduce direct, unchecked supervisor hour entry as the default path. The old `HoursForwarded` collection/"Forward to Accountant" button was removed as dead code (nothing downstream read it, left over from before Accountant lost payroll ownership) — don't recreate it.
- **HR onboarding no longer has an approve/reject action.** It briefly did (an "Approve"/"Reject" icon pair on `hr_dashboard.dart`'s Submitted tab flipping `EmployeeDetails.status` to `approved`/`rejected`), but since HR is the *only* actor who ever creates or edits onboarding records — there's no separate submitter to check — that second-approval step was pure friction and was removed (`_approveEmployee`/`_rejectEmployee` deleted). `status` now only ever moves `draft` → `submitted` via the wizard's own Submit button. A few pre-existing records may still carry legacy `approved`/`rejected` values from before this removal; `_buildStatusBadge` still renders them for display, but nothing sets them anymore. Don't re-add an HR-side approve/reject action without checking this is actually wanted.
- **`hr_dashboard.dart`'s employee stream must never use `.orderBy()`.** It did briefly (`.orderBy('createdAt', descending: true)`), which silently drops any `EmployeeDetails` doc missing that field from the query results entirely (a documented Firestore behavior, not a bug in Firestore) — this hid at least one real employee record. Sorting is done client-side in `_getEmployeeDetailsStream()` instead (docs without `createdAt` sort last, but are never excluded). If you need a different sort order, keep it client-side rather than reaching for `orderBy`.
- **`hr_dashboard.dart`'s status filter has three states**: `all` (default) / `draft` / `submitted` (the latter bucket also covers any legacy `approved`/`rejected` doc — anything that isn't `draft`). Don't assume it's still the old two-way Draft/Submitted toggle.
- **A single malformed `EmployeeDetails` doc must never blank out `PayrollService.getEligibilityPreview()`/`generateRun()` for every other employee.** Both methods iterate every doc in a per-record `try/catch` specifically so one bad/legacy record is reported as its own skipped/incomplete entry (with a reason) instead of an exception aborting the whole loop and returning an empty list. If you add new per-employee processing to either method, keep it inside that per-record try/catch — don't wrap only the outer loop.
- **NHIF was replaced by SHIF/SHA** — `StatutoryDocuments.shifNumber`/`shifConfirmationUrl` and `BenefitsInsurance.shifDependants` are the current field names everywhere (model, onboarding UI, HR dashboard, Excel export). Never reintroduce "NHIF" in a label or a newly-written field. Records onboarded before this rename may still have data under the legacy keys `nhifNumber`/`nhifConfirmationUrl`/`nhifDependants` — every reader falls back to those keys, so don't remove the fallback without a data migration.

## Running the app

Standard Flutter workflow: `flutter pub get`, `flutter analyze`, `flutter run -d chrome` (or `-d windows`/`-d macos`). Firebase config is already wired (`lib/firebase_options.dart`, `firebase.json`, `.firebaserc`); no additional setup needed for local development against the existing Firebase project.
